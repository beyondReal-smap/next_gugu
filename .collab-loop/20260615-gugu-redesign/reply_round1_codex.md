[Summary]
- 방향성은 맞지만, MVP 기준으로는 서버형 리텐션과 기능 과잉이 섞여 있어 설계가 너무 넓습니다. 정적 export 기반에서는 로컬 후크만으로도 충분히 리텐션 MVP를 만들 수 있고, 리그/푸시/동기화는 후속으로 빼는 편이 낫습니다.

[Findings]
- P1. 정적 export와 서버 의존 후크가 한 문서에 섞여 있습니다. `docs/DESIGN_v1.md:18-20,57-64`, `next.config.js:3-9`
  - 정적 export 유지라면 `스트릭/데일리 골/레벨/마스터리 맵`까지만 MVP 리텐션이 성립합니다.
  - 위클리 리그, 푸시, 크로스기기 동기화는 서버가 있어야 의미가 있으므로 MVP에서 빼야 합니다.
  - `next start` 전환은 단순 선택이 아니라 배포 파이프라인 변경입니다.

- P1. 타깃과 톤이 서로 충돌합니다. `docs/DESIGN_v1.md:9-12,45-49`
  - 초등 3~6 + 성인 두 집단을 동시에 잡으려면 인터페이스 언어가 분열됩니다.
  - 다크 우선은 성인에게는 좋지만, 초등 학습 동기에는 차갑게 느껴질 가능성이 큽니다(추정).
  - 1차 타깃을 상단 학년 또는 중등 저학년으로 좁히고, 성인은 별도 모드로 두는 편이 맞습니다.

- P1. MVP 범위가 과합니다. `docs/DESIGN_v1.md:15-21,25-32,35-42,62-64`
  - 한 번에 `4모드 + 리그 + 데일리 챌린지 + 통계 + 업적 + 맵 + 적응형 출제`를 넣으면 구현보다 운영 복잡도가 먼저 커집니다.
  - 리텐션 ROI가 높은 것은 `데일리 골/스트릭/레벨/즉시 피드백/간단한 마스터리`입니다.
  - `위클리 리그`, `공유`, `풍부한 통계 대시보드`는 후순위입니다.

- P2. 상태관리/차트/PWA의 추가 도입이 과합니다. `docs/DESIGN_v1.md:53-60`, `package.json:15-23`
  - 현재 저장소는 이미 `framer-motion`, `next-pwa`, `Capacitor`가 들어가 있어 새 스택을 더 얹는 순간 복잡도가 더 커집니다.
  - `Zustand`는 단일 학습 게임의 MVP에 필수는 아닙니다.
  - 차트도 처음부터 `Recharts`보다 `SVG`나 아주 단순한 컴포넌트로 시작하는 편이 낫습니다.

- P2. SRS는 이 도메인에서는 축소해서 써야 합니다. `docs/DESIGN_v1.md:17,31`
  - 구구단은 항목 수가 적어서 정교한 스페이스드 리피티션보다 `틀린 문제 가중치 + 최근 오답 재출제`가 비용 대비 효율이 높습니다.
  - 테이블 단위 마스터리와 오답 우선 출제만으로도 학습 효과는 충분히 낼 수 있습니다.
  - 완전한 SRS 모델은 과설계로 흐를 가능성이 큽니다.

- P2. 브랜드/광고 정체성 정리가 빠져 있습니다. `app/page.tsx:36-101`
  - 현행 화면은 SMAP 배너와 구매 유도 흔적이 강합니다.
  - 새 제품이 “구구단판 듀오링고”라면, 광고/브랜드 혼선을 문서에서 먼저 잘라야 합니다.
  - 이 부분을 방치하면 모던한 학습 앱 톤이 깨집니다.
